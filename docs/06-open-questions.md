## 調査の運用

- 各Qの結論は本ファイルに追記し、設計変更が生じたら該当ADRを改訂（Supersede）する。
- Q1（日次アンカー）と Q4（エラー課金）は数字の信頼性に直結するため、結果が出るまで
  関連指標（未計上・乖離）には「暫定」ラベルを付けて表示する。

### Q1 の検証手順（2026-07-20 時点の前提整理）

定額サブスクリプションなど、請求書から日次×モデルの金額を確定できない環境では
検証できない。必要な粒度を提供する課金アカウントを検証用に用意する。

**現時点で確定している前提**:

- 単価表は API から取得できず**自前管理**（[02-data-model](02-data-model.md)）。したがって
  単価の誤りは自前計算に系統誤差として乗る。
- CC クライアント推計（`cost.total_cost_usd`）は別系統の実装だが、**同じ単価表の誤りを
  共有しうる**とは限らない一方、**独立性の程度は未検証**。つまり現状の二点照合が
  系統誤差に対してどれだけ有効かは分かっていない。

**確認すること**（上から順に。1つでも取れれば校正の質が上がる）:

1. console の Usage/Cost 画面に **金額**が日次×モデルで出るか（出れば校正が月次→日次化）
2. 金額が出なくても **トークン量**が日次×モデルで出るか
   → 出るなら**トークン量での突合**が成立する。これは金額突合より強い場面がある:
   **単価表の誤りと token 集計の誤りを分離できる**（金額だけの突合では両者が混ざる）
3. CSV エクスポートの有無と粒度
4. Usage & Cost Admin API が対象のアカウント種別で利用可能か・レスポンスの粒度

**取得したら記録すること**: 画面/APIの粒度そのものに加え、**同一期間について
`metsuke today --json` / `metsuke view period --month` の自前計算値**を並べて残す。
突合の生データが無いと、後から係数を導出できない。

**結果による分岐**:

- 1 が取れる → §5 校正プロトコルを日次化。J6 の検知遅延が30日→1日
- 2 のみ → 校正を「トークン量アンカー＋自前単価表」に再設計（単価の誤りは
  別途 SCD2 の `price` 表の履歴で追う）
- どちらも不可 → 校正プロトコルを「SDK推計との整合」ベースへ変更し、
  **外部アンカー不在を明示的な限界として [05-risks](05-risks.md) に格上げする**

- **Q18 = jq 呼び出し集約で決着**（2026-07-20・同条件40回中央値）:
    - **原因**: 処理内容ではなくプロセス起動が支配項だった。変更前は通常描画1回で `jq` を
      **27回** fork し、実測約3ms/回（bash 起動約2.6ms）だけで約84msとなる。発見時の
      壁時計は約87ms、今回の同条件再計測は中央値 **70.3ms** だった。
    - **対策**: stdin の解析を1回、`state.json` の抽出・色判定・stale判定・TTL計算・
      prompt行生成を1回へ集約した。時刻は `date +%s` で1回だけ取得し、数値の表示書式は
      bash 組み込み `printf` に維持した。センサーサンプル生成の `jq` は書込時の1回だけ残した。
    - **結果**: PATH シムによる実測は通常 **2回**、センサー書込時 **3回**。同条件40回の
      壁時計中央値は **70.3ms → 23.7ms** となり、R1（50ms以内）を満たした。同一の
      `state.json` と stdin による変更前後の出力は ANSI escape を含むバイト列で一致した。
      副次効果として、stdin がオブジェクトでない場合の ctx 表示も `%` から仕様どおりの
      `0%` に修正された。
    - **機械検証**: 壁時計はマシン負荷で揺れて flaky になるため採用せず、根本原因を直接押さえる
      `jq` 実行回数を代理指標とした。通常3回以下・センサー書込時4回以下（実測値+1）を
      `tests/test_hotpath.py` で強制し、R1 の退行を安定して検知する。クローズ。

- **Q5 = 全量集計で分布確定、規則6の精緻化候補を発見**:
  中断イベントは、**(a) 参照先assistant行がなく完全未計上、(b) `stop_reason`付きで
  outputが信頼できる、(c) `stop_reason=null`でoutputがプレースホルダ**の3群に分かれた。
  (a)が約4割、(b)が約3分の1、(c)が約2割だった。**現行規則6は is_interrupted 行の output を一律 NULL 化する
  ため、(b) の信頼できる実数まで捨てている**。ただし `v_unaccounted` は「参照先 request
  が丸ごと不在」の集合（＝(a)のみ）を対象とし、(b)(c) の found 行は含めないため、
  (b) の output を復元しても **v_unaccounted との二重計上は発生しない**（影響は
  `ingest.py` の 2箇所のみ）。金額影響は小さい。
  **決定・実装済み**（2026-07-19・ユーザー承認）: 規則6を「`stop_reason` が null の
  中断行のみ output を NULL 化」へ精緻化（marker UPDATE ＋ upsert 競合ハンドラの2箇所・
  `is_interrupted=1` は真の事実として全行で維持・PARSER_VERSION=7）。upsert の順序ハザードは
  `COALESCE(excluded.stop_reason, request.stop_reason)` で吸収し、golden fixture で
  「marker が重複行の前/後どちらで発火しても output 温存」を回帰テスト。rebuild済み実台帳で
  信頼できるoutputが復元され、stop_reason=null群はNULLのまま維持されることを確認。クローズ。

- **Q14 = 分解済み・unknownは複数現象の混合と判明**:
  unknownはread過大（増加）とread喪失がほぼ半々で、同じ「破れ」として扱うこと自体が
  判断を曇らせていた。実運用台帳で次を比較し、設計根拠を確定した。

  - **hook依存causeのカバレッジ**: compaction/config_changeは計装開始前へ遡及できない。さらに
    cause判定の相関サブクエリで外側時刻を修飾しておらず、上限のない窓になっていた。修正後は
    誤ラベルが大幅に減り、分類カバレッジは約4割から約3割へ下がった。これは劣化でなく水増しの是正。
  - **符号分離**: read喪失だけが再構築費を生み、read過大はキャッシュが増えた事象である。
    符号で分けるだけで「損失unknown」は約半分になり、V3の問いと表示が一致した。
  - **5m TTL**: 1h規則だけでは説明できない喪失の一部を、直前が5mのみ、実待機が300秒超、
    全消失、生存中の1h書込なし、の4条件で再分類した。単純な開始時刻差では長いrequestを
    idleと誤認するため、終了時刻からの実待機を使う。カバレッジは数ポイント回復した。
  - **活動中の喪失**: 内容・総contextがほぼ変わらないtool loop途中でも全再writeが起きた。
    idle TTL、context縮小、巨大入力、段階的な1h失効では説明できず、prompt cachingがbest-effortで
    ある挙動と整合する。provider側evictionかクライアント側設定かはtranscriptだけでは区別しない。
  - **増加側**: 約8割は直前outputが次のreadへ現れるのに恒等式がoutput項を持たないための会計上の
    偽陽性で、残りは大容量contextの再アタッチだった。どちらも損失ではない。恒等式へoutputを
    無条件追加すると正常遷移に大量の偽喪失が生じるため、補正は既検出の増加行へ限定しなければならない。
    金額判断はほぼ変わらないため、今回は符号分離を優先し補正自体は見送った。
  - **read二重計上**: 報告値が直前prefixの約2倍になる事象は、増加と喪失を1件ずつ作る。
    設計時サンプルでは増加の約1割・喪失の約6%を説明した。ingest合算ではなく原本の報告値であり、
    実課金かは外部請求アンカーなしに判定しない。

  V3は喪失・増加を分離し、hook計装前の割合を動的注記する。4条件とスコープ修正は否定テストで固定し、
  分類不能を無理にユーザー行動へ帰属させない。

- **Q15 = 「回避可能」の大半は回避不能と判明**:
  V3が回避可能と提示していた放置起因の喪失は、実運用サンプルでは**約9割が再開後の実作業に
  償却**され、行動で消せるのは全再構築費の4〜7%だった。

  | 層 | 相対的な規模 | 回避可能性 |
  |---|---:|---|
  | 増加（損失でない） | 費用影響はごく小さい | 対象外 |
  | 喪失・活動中 | 大きな費用群 | **不可**（best-effort caching） |
  | 喪失・5〜60分放置 | 中規模だが大半はTTL内失効 | **ほぼ不可** |
  | 喪失・1h超放置 | 最大群 | TTL失効自体は不可避。再開するかだけが選択 |

  再開後の償却判定を30分・2時間・6時間の窓で比較すると、窓を広げるほど非償却分は減るが
  結論の符号は変わらなかった。開始時刻差と終了時刻からの実idleでも層は変わらず、計測基準の
  混同は結論に影響しない。残るレバーは巨大contextをタスク境界で持ち越さないセッション衛生だが、
  compactの要約writeと忠実性低下を伴うため、線形換算した節約上限を目標値にはしない。
  この結果を受け、V3の区分軸を「放置/活動中」から**回避可能性**へ切り直した。

- **Q16 = 誤診と判明、現行の実装が正しい**:
  基準日を「その日全体にrequestがあるか」で採用し、同時刻までの累積が0の日も分母に含める
  非対称を疑ったが、同曜日の同時刻までの典型支出という定義では0も正当な標本である。

  代案の `partial>0` と全セル比較すると、両方式が値を返す箇所は完全一致した一方、代案は
  基準日不足により約3分の1の出力を失った。値を改善せず欠測だけを増やすため採用しない。
  履歴が少ない時間帯では分母が小さくnullへ縮退し得るが、過小な数値を出すより正しい。
  教訓は、実装の見た目の対称性ではなく指標定義からゲートの妥当性を判断すること。
