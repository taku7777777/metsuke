# 07 — ビュー台帳: 判断支援ビューの対象管理

気づきの動線の中段（pull型の精査層）を埋めるビュー群の**唯一の管理場所**。枠の規範は
[ADR 0007](adr/0007-decision-support-views.md)。原則: **「答える問い・見るきっかけ・
次の行動」を1行で書けない画面は作らない**（1画面=1問い）。

動線の全体像: statusline（常時・高コスト色/詳細リンク）→ hooks/緊急通知（瞬間）→ 週次アナリスト（push・仮説）→
**判断支援ビュー（pull・精査）** → trace（深掘り）→ raw/Datasette（全事実）。
良し悪しの最終判断は人間。システムは外れ値・変化を「気づきやすく」提示するまで。

実装状況（2026-07-21）: **V1〜V4 すべて実装済み** — 現行は
`metsuke view <period|trend|cache|dist>`の自己完結HTML。Stage 8で、問いとquery定義を変えずに
動的ローカルdashboardへ通常の提供面を移す（[ADR 0011](adr/0011-local-dashboard.md) /
[08-dashboard](08-dashboard.md)）。
実体は `src/metsuke/viewgen/`（`render.py` がマークアップを出力する唯一のモジュール）。
生成のたびに`hook_event(kind='view_html_generated')`へ記録される。Stage 8実装後の撤退判定は
これに`dashboard_view_opened`を加え、自動preview等は除外する。

## 採用（優先度順）

| # | ビュー | 答える問い | 見るきっかけ → 次の行動 | 主データ源 | PII面 |
|---|---|---|---|---|---|
| V0 | trace（実装済み） | この1プロンプトはなぜこの値段・この時間か（J1/J5） | statuslineの高コスト値・上位ビューの外れ値 → raw_path で原本へ | ledger＋transcript本文 | 本文込み（ADR 0006） |
| V1 | [period（期間）](views/v1-period.md) | この期間、**どのプロンプト/セッション/プロジェクトにいくら**使い、外れ値はどれか（J1/J3） | 週次レポの根拠確認・請求感覚とのズレ・statusline の暴走感 → 外れ値の行から trace へ降りて背景込みで精査 | v_request_cost・prompt・session（期間述語で絞ってから集約 — v_prompt_cost は期間集計源にしない） | redact済み prompt 冒頭のみ（本文読み出しなし） |
| V2 | [trend（推移）](views/v2-trend.md) | 週次の費用構成と行動指標は**変わったか** — 変化はどの軸・どの施策期間に対応するか（J3） | 週次レポの前週比主張・未判定 marker の検証 → 変化点の期間を V1 で深掘り・費目異変は V3 へ | v_request_cost・session・prompt・v_daily・v_cache_identity・marker・regime_event | 数値のみ |
| V3 | [cache（キャッシュ健全性）](views/v3-cache.md) | キャッシュ投資（write費）は**回収されているか** — 破れ⚡はいくらの損で原因はどこか（J5） | 領収書の cache_w 違和感・V2 で費目異変・月次ベンチ → 該当セッションを trace で精査 | v_cache_identity・v_request_cost・v_context_overhead | 数値のみ |
| V4 | [dist（プロンプト分布）](views/v4-dist.md) | コスト/コンテキストの分布は**どんな形**で、高コストはどこに住み正体は何か（J1） | 週次レポの分布主張の裏取り・月次請求突合の儀式・領収書ショックの正常性判定 → 帯/上位10%の代表例を explain、同一期間を持って V1 へ | v_request_cost・prompt・session | 数値のみ |

対象確定（2026-07-18 FB）: **V1〜V4 の4枚**。V1 の軸はプロンプト/セッション/プロジェクトの
3軸ミニマム（モデル・日の軸は V2、分布の形は V4 に任せる）。統計の一意所有:
**集中度=V1・時系列=V2・効率(blended単価等)の断面=V3・分位点の断面=V4**。
分位点・blended単価の**推移**は V2 が持つ（定義は V4/V3 と共有 — 断面はV4/V3・推移はV2）。詳細（目的・提供価値・UC・
変更方針）は views/ 配下の各ファイルが正。実装済み画面のtrace降下は現在
**`metsuke explain/trace <id> --html` のコマンド併記**。Stage 8ではprompt/session行をリンク化し、
未生成traceを遅延生成して対象アンカーへ遷移する（ADR 0011）。

## 作らない（却下・時期尚早）

| 対象 | 理由 |
|---|---|
| 汎用・外部公開dashboard / リアルタイム監視 | J2（ペーシング）はstatuslineで充足。pull探索用のloopback限定dashboardはADR 0011で採用 |
| タスク種別の効率ビュー（J4単体） | outcome/marker のラベルが未成熟。コスト加重カバレッジ80%達成時に V1 第4軸として再検討 |
| 計測ヘルスの画面化（J6） | v_health（CLI）で足りる。画面を増やす価値なし |
| セッション一覧ビュー単体 | V1 のセッション軸グルーピングで兼ねる |
| モデル切替頻度ビュー | V3 の cause=model_switch 件数で充足 |
| 背景機能 / query_source / effort 軸 | OTelタップ（2026-07-17）導入直後で蓄積不足。4週分溜まったら V1 insight 1行で再検討 |
| 価格ウェイト式実効トークンの分母採用 | 分散が消え blended単価のキャッシュ診断力が失われる（2026-07裁定）。「請求異常検知」の問いが立った時のみ再検討 |
| V2 の管理図・同曜日比・複数窓移動平均 | n=1運用・右裾分布で偽シグナル過多／各曜日の標本不足。管理図のみ8週蓄積後に再検討（7日移動平均1本は採用） |

## 運用

- 追加はこの台帳に行を足し「問い・きっかけ・行動」を書け、遷移表の from（実在の上流イベント）に
  紐づくことを確認してから。詳細定義は views/ 配下に1ビュー1ファイル。
- 週次アナリストレポは主張ごとに根拠コマンド（`metsuke view <name> --week last` 等）を併記する
  （pull型ビューのトリガーを push 側が製造する。ADR 0007 の「対にする」の具体化）。
- 自動実行ワークロード（sandbox 等）の判別は project 名ベースの読み出し時導出で1箇所に定義し、
  中央値・分位点系の指標は「対話のみ」を併記する（除外はしない — 総額の帳尻を守る）。
- 各ビューはCLI生成記録（hook_event kind='view_html_generated'・payloadにビュー名）と
  `dashboard_view_opened`で利用実測し、
  四半期棚卸しで使われていない行を削除する（ADR 0007 の撤退基準）。

## 実運用ループの摩擦ログ

「V3の外れ値 → trace → mark → 行動 → 翌週V2で前後比較 → verdict」を実際に回して出た詰まり。
**ここに溜めて改修の入力にする**（推測ではなく実走で出たものだけを書く）。

### 第1周（V3 cache）

到達点: V3生成 → ttl_expiryの最大外れ値を特定 → `metsuke trace` で背景確認 →
**マクロ帰属まで到達し、そこで「介入に値する損失がほぼ無い」と判明**（[06-open-questions Q15]）。
`metsuke mark start` は**実施しない**。打つべき施策が無いという結論だったため、marker を作れば
実体のない記録になる。**打つべき施策が無いという結論自体がこの周の成果**であり、marker を作らないのは正しい。
ただし裏返しとして、**`mark start` → 行動 → 翌週 V2 前後比較 → `verdict` の下流半分は依然として一度も
通っていない**（下記の摩擦はすべて view→trace の上流で出たもの）。閉ループ検証は未達のまま。

| # | 摩擦 | 具体 | 対処案 |
|---|---|---|---|
| F1 | **views.sql の反映漏れが黙って起こる** | `metsuke view` は `connect_readonly()` で VIEW を張り直さない。直近2コミットが `v_cache_identity` を変更していたため予防的に `metsuke sync` を先に通し、その後 `sqlite_master` の定義が `views.sql` と一致することを確認した。**この周に実際に罠が発動したかは未観測**（sync前の状態を見ていない。前セッションが実DBで計測値を出している以上、定義は既に最新だった公算が高い）。問題は**「旧定義でも黙って尤もらしい値が出る」構造と、回避が人間の記憶に依存している**こと（RUNBOOK §4 に記載はある） | 読み出し時に `sqlite_master` の定義と `views.sql` のハッシュを照合し、不一致なら**警告を出す**（自動再作成はしない — 読み取り専用接続の不変条件を壊さないため） |
| F2 | **V3→V0 の受け渡しでアンカーが落ちる** | 上位表の command 列は `metsuke trace <session_id> --html` のみ。大規模trace内に同原因が複数散らばり、**どれが表の対象行か判別できない**。金額も trace 側に出ない | 上位表の command に発生時刻（または request_id）を持たせ、trace 側で該当行を強調する。ADR 0007 の「リンク化は後回し」方針は維持しつつ、**持ち物だけは落とさない** |
| F3 | **ビューが HTML のみで、端末に要約が出ない** | `metsuke view cache` の標準出力はファイルパス1行。insight・上位表の中身をブラウザ以外から読めず、CLI上で判断を続けられない | `--text`（insight＋上位表を端末へ）を追加。生成HTMLは PII 面の制約（ADR 0006 §8）があるが、V3 は数値のみなので text 出力の追加リスクは無い |

いずれも **ビューの中身ではなく動線の欠落**。

**対処状況（2026-07-20）**:

- **F2 = 解消**。`metsuke trace --focus <request_id>` を追加し、V3 喪失上位表の command に
  載せた。あわせて ⚡ラベルに**原因・再構築費・時刻**を出し、
  focus を使わなくても金額で目視照合できるようにした。**金額書式が V3 と1桁でも違うと
  照合という目的を果たさない**ため、`render.money` と文字列一致することをテストで固定している
  （レビューでは丸め桁の不一致も実機で検出・修正させた）。
  再構築費の式は `v_request_cost.cache_write_usd` に単一化した。
- **F1 = 保留（危険度は下方修正）**。F2 で `views.sql` を変更した直後に `metsuke view` を叩いて
  実測したところ**成功した** — launchd の自動 sync が既に新定義を適用済みだったため。
  **露出窓は「編集から次の sync tick まで」**であり、当初の想定より短い。ガード（定義ハッシュの
  照合と警告）の価値は残るが優先度は下げてよい。
- **F3 = 見送り**。`--text` は端末専用エージェント側の都合であり、ブラウザで見る実運用では
  価値が低い。必要になってから足す。
