# V4 dist（プロンプト分布ビュー） — コストの形と高コストの正体

- **答える問い**: プロンプトコストの分布はどんな形で、**高コストはどこに住み、その正体は何か**（J1）
- **3秒行**: 「≥200kコンテキスト帯が件数X%でコストY%」「cost上位10%: req中央値A（残りはB)・観測スパンC分」
- **次の精査コマンド**: 帯・cost上位10%の代表例に `metsuke explain <prompt_id> --html` を併記
  （リンクは生成しない。コピー可能なコマンド文字列を表示）。個体の一覧 → `metsuke view period`（V4と同一の期間引数で）

## 目的

個別の外れ値ではなく**分布の形**で自分の使い方を理解し、**運用ポリシー**
（セッション分割・/compact・委任・着手前見積り）を決める根拠を与える。
3表は1つの問いの分解: **全体・プロジェクト別分位点で形を確認 → コンテキスト帯で高コストの住所を特定 →
上位10%比較でそのプロファイルを把握**する。
V1が「どれか」に答えるのに対し、V4は「これは異常か、構造か」に答える。
**分位点の唯一の所有者**（全体=p50/p90/p95/p99、プロジェクト別=p50/p90/p95。V1には出さない）。

## 提供価値

1. **相場観（J1見積り）**: 分位点の相場（普通の依頼・重い依頼の金額帯）が身につき、
   領収書ショックの正常/異常判定と着手前見積りができる
2. **金の住所**: コンテキスト帯×コストシェアで「金の過半は≥200k帯に住む」型の構造事実が出る
   → 200k以上・500k以上のコンテキスト運用の妥当性・分割/compactポリシーの判断材料
3. **高コストの正体**: cost上位10% vs 残りの req数/観測スパン/中断率/委任費率の対比で、
   「外れ値を個別に潰す」から「長ループ運用を設計する」への発想転換を促す
4. **プロジェクト別の1依頼相場**: p50/p90の差でプロジェクト間の相場の違いを把握

## 画面構成と読み方

- insight（3行構成・主役から順に）:
  1. 「≥200k: 件数X% → コストY%」の非対称ペア
  2. 「cost上位10%: req中央値A（残りはB）・観測スパンC分」
  3. コスト分位点 p50/p90/p95/p99（全体と「全体(対話のみ)」）
- コスト分位点×プロジェクト表: 期間内cost降順の上位7project＋その他＋全体＋**全体(対話のみ)**:
  project / n / 合計(bar) / p50 / p90 / p95
  （project行のみ n<20 では p90/p95 を非表示 — 不安定な比較を避ける設計閾値）
- ピークコンテキスト帯表（**<200k / ≥200k–<500k / ≥500k の3帯**・各帯を総量/対話のみの2行で表示。
  シェアの分母: 総量行=全母集団、対話のみ行=対話のみ母集団 — 件数・コストとも）:
  帯 / prompts / 件数シェア / コスト(bar) / コストシェア / **$/prompt中央値** / 主モデル /
  代表例（帯内cost上位2件の `metsuke explain <prompt_id> --html`・同値時はprompt_id昇順）
- **cost上位10%のプロファイル表**: cost上位10% / 残り90% / 全体 ×
  n / req中央値 / 観測スパン中央値 / **中断率** / **委任費率** / req p90 /
  代表例（上位10%行のみ・cost上位2件の `metsuke explain <prompt_id> --html`）
  — 注記は観測事実のみの機械生成文
  （例: 「上位10%は残り90%よりreq中央値がX倍・観測スパン中央値がY倍・委任費率がZpt高い」— 「迷走」等の断定はしない）

指標の定義:

- **母集団** = 期間内の prompt_id 帰属済みプロンプト。**総支出ではない** —
  帰属カバレッジ = 期間内 `SUM(帰属済みcost_usd) ÷ SUM(v_request_cost.cost_usd)` をヘッダに表示
  （未知価格 cost_usd NULL のrequestがある期間は件数を別途警告）
- **ピークコンテキスト** = promptごとに、`v_request_cost` の `agent_id IS NULL` かつ `prompt_id IS NOT NULL` の
  各requestの `input_tok+cache_read_tok+cache_w5m_tok+cache_w1h_tok` の **MAX**。
  メインレーンrequestを持たないpromptは帯表から除外し件数を注記
- **cost上位10%** = 期間内promptをcost降順に並べた上位10%（件数基準・端数切上げ・同値はprompt_id順）
- **中断率** = `MAX(is_interrupted)=1` のprompt数 ÷ prompt数
- **委任費率** = `agent_id IS NOT NULL` のcost ÷ prompt総cost
- **観測スパン** = prompt帰属requestの開始時刻幅 `MAX(ts)−MIN(ts)`（分）。
  処理時間ではない（単一requestでは0・人間待ちや並行agentを含み得る代理指標）
- **対話のみ** = 台帳共通のproject名分類で自動実行ワークロードを除いた母集団（V4固有の判別は持たない）
- 帯境界: **200kは長コンテキスト割増の価格境界**。**500kは高コンテキスト域を分ける運用上の固定境界**
  （設計時裁定 2026-07-18・分布が変わったら再検討）。境界値は下限含み（≥200k, ≥500k）

読み方の注意:

- 分位点は**混合分布**（自動実行バッチ＋対話）。判断は「対話のみ」行で行う
- 帯は固定境界なので再生成間で比較できる（分位点ベースの動的区切りは不採用）
- ≥1M件数を帯表の補助注記に出す（1Mコンテキストモードの利用実態の確認用）
- 週次レポの集中度主張（「上位N件でX%」）の裏取りは**V1の領分**。V4で裏取りするのは
  分布・帯・プロファイルの主張のみ

## ユースケース

### UC1 週次レポの分布主張の裏取り
- きっかけ: レポの「≥200k帯がコストの7割」「上位10%はreq中央値26の長ループ」等の
  分布・プロファイル主張（レポにコマンド併記。遷移表: 週次レポ→V4）
- コマンド: `metsuke view dist --week last`
- 見る値: コンテキスト帯表のコストシェア・プロファイル表
- 判断: 主張は正しいか。高コスト群のプロファイルは多req型・長スパン型・高委任型のどれか
  （「個別外れ値か構造か」の最終確認は同一期間のV1で行う）
- 行動: 構造なら分割・委任・見積りの施策を
  `metsuke mark start --category <category> --hypothesis "…" --expected "…"`（判断投入CLI・想定）で1本立てる（V2で前後検証）

### UC2 月次ポリシー見直し（請求突合と同時）
- きっかけ: 月1の請求突合の儀式（遷移表: 月次請求書→V1＋V4同時）
- コマンド: `metsuke view dist --month <YYYY-MM>`（当月を確認・記録した後に前月分を生成して目視比較。
  生成物は同一ファイルを上書きするため同時表示はしない）
- 見る値: コンテキスト帯表のコストシェアと$/prompt中央値の前月との差・≥1M件数注記
- 判断: 200k以上・500k以上のコンテキスト運用は妥当か。帯シェアが育っていないか
- 行動: 育っていればセッション分割 / /compact励行を `metsuke mark` で施策化。妥当なら現状承認

### UC3 領収書ショックの正常性判定
- きっかけ: 高額領収書に驚いた直後（遷移表: 領収書→V4→V0）
- コマンド: `metsuke view dist`
- 見る値: 領収書記載額を分位点と目視比較（どの分位帯か）・cost上位10%の典型プロファイル
- 判断: 異常ではなく「p99帯の型の仕事」（多req・長実働クラス）かどうか
- 行動: 型の仕事なら着手前見積りの習慣（J1）に接続。異常なら領収書の
  `metsuke explain <prompt_id> --html` で個体精査。頻度の疑問はV1へ（同一期間で）

## 利用実測と撤退

生成時に `hook_event.kind='view_html_generated'`・`payload_json` の `$.view='dist'` を記録し、
四半期棚卸しで生成0件ならこのビューを削除する（ADR 0007。現行の週次レポは
**コマンド併記のみ**で自動生成はしないため、全実行を利用件数として扱う）。
Stage 8移行後はdashboardのview表示イベントを利用分母とし、静的HTML生成件数だけで撤退判定しない
（[ADR 0011](../adr/0011-local-dashboard.md)）。

## 設計時サンプル

恒久定義ではなく、設計判断の根拠となった実測値。再生成後の画面とは一致しない。

- コストは強いロングテールで、上位10%が帰属済み総額の約4分の3を占めた
- ≥200k帯: 件数は約2割だがコストは約7割（中間帯だけでもおよそ半分）
- cost上位10%は残り90%よりrequest数が1桁多く、観測スパンも桁違いに長かった。
  委任併用prompt率は約4割（費率ではなく委任ありpromptの件数率）
- ピークコンテキストはモデル群によって二峰性を示し、標準帯と拡張コンテキスト帯を分ける必要があった

## 現行プロトからの変更方針（2026-07-18裁定）

1. insightの主役を「≥200k: 件数→コスト」の非対称ペアに差し替え、プロファイル（req対比）を2行目に昇格。
   分位点は3行目に降格
2. コンテキスト帯を <200k / ≥200k–<500k / ≥500k の3帯に再定義（150–200k帯は旧上限の遺物のため削除。
   **未決事項「帯区切り」はこれで裁定・クローズ**）。$/prompt中央値列・≥1M件数注記を追加
3. 分位点表・帯表に「対話のみ」行を追加 — 除外ではなく併記。判別は台帳共通のproject名分類のみ
   （モデル群ヒューリスティックは使わない — 他ビューとの分類一致を優先）
4. プロファイル表に中断率・委任費率の2列を追加し、表名を「cost上位10%のプロファイル」に変更。
   「実働」は「観測スパン」に改名
5. 帯と上位10%行に代表例2件（cost上位・tie-breakはprompt_id昇順）の `metsuke explain <prompt_id> --html` を併記
   （IDは8文字プレフィックス表示 — CLIがプレフィックス解決）
6. 固定文の注記を観測事実のみの機械生成文に置換（断定的解釈は書かない）
7. 週次レポテンプレへの `metsuke view dist` コマンド併記は**依存事項**（レポ側実装が受入条件。
   台帳の運用規範に記載済み）
8. CLI要件: `--week last` / `--month` / `--from--to` を受ける
