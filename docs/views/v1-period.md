# V1 period（期間ビュー） — 支出の帰属とtraceへの降下ハブ

- **答える問い**: この期間、どのプロンプト/セッション/プロジェクトにいくら使い、外れ値はどれか（J1/J3）
- **3秒行**: 「選択中の軸の上位K件で、その軸の集計対象額のX%」— 集中か分散か、外れ値はどの行か
- **次の精査コマンド**: プロンプト行 `metsuke explain <prompt_id> --html` / セッション行 `metsuke trace <session_id> --html`
  （リンクは生成しない。コピー可能なコマンド文字列を表示）。分布の形の疑問 → `metsuke view dist`。変化の時期の疑問 → `metsuke view trend`

## 目的

期間内の支出を**個体に帰属させるランキング面**。「どこに使ったか」を3軸で答え、
疑わしい1行からコマンド併記でV0（trace/explain）へ降りる主要な入口になる。
統計の一意所有: **V1は集中度と個体の特定（どれか）**、分布の形（どんな形か）はV4の専有。

## 提供価値

1. **帰属**: 上位ランキング＋累積%で「集中か分散か」「外れ値はどれか」が3秒で出る
2. **降下の入口**: プロンプト行・セッション行に実行可能なコマンド文字列を併記。
   プロジェクト行はセッション軸への絞り込み（`--project`）で個体へ降りる
3. **上流との突合**: 週次アナリストレポ・月次請求・statuslineの数字を**同一期間**で検算する場
   （期間プリセット `--week last` / `--month <YYYY-MM>` / `--today` が成立条件）

## 画面構成と読み方

3軸タブ（UI順: **セッション=デフォルト** / プロンプト / プロジェクト）。
典型的な絞り込み順はプロジェクト → セッション → プロンプト（マクロ→ドリル）。

| 軸 | insight | 主な列 |
|---|---|---|
| セッション | 上位10件で全request総額のX% / $1以上はN件（全M中）でY% | コスト(bar)・累積%・開始・期間・project・prompts・req・**$/prompt**・**平均ピークctx**・command |
| プロンプト | 上位10件で$X（帰属済み総額のY%）/ 最大$Z(project) / 中断プロンプト支出$A(N件) / 帰属不能$B(C%) | コスト・累積%・日時・**期間**・project・prompt冒頭・req・agent・費目ドット・command |
| プロジェクト | 上位3件でX% / 全N中$1未満M件 | コスト(bar)・share・sessions・prompts・req・**$/prompt**・**平均ピークctx** |

指標の定義（分母を固定する）:

- **期間集計の共通規則**: 全軸とも「まず `v_request_cost` を期間述語で絞り、その結果を
  prompt/session/project単位に集約する」。`v_prompt_cost` は全履歴プリ集約（`ts=MIN`）のため
  期間ビューの金額集計源には使わない（期間境界をまたぐプロンプトの誤計上を防ぐ）
- **セッション/プロジェクト軸の分母** = 同一期間の `v_request_cost` 全行の記録済みコスト（ヘッダ総額と同一の期間述語）
- **プロンプト軸の分母** = 同一期間の prompt_id 帰属済み記録済みコスト。
  **帰属不能** $B = `prompt_id IS NULL` の記録済みコスト、C% = B ÷ 全request記録済みコスト。insightに常時明示
- **中断プロンプト支出** $A = 期間内requestで `MAX(is_interrupted)=1` となったプロンプトの期間内 `cost_usd` 合計。
  「損失」ではない（中断前の成果が使われた可能性があり、中断行の output は未計上のため過小）。
  未記録分は `v_unaccounted`（入力側下限推計）が別指標として存在する
- prompt冒頭 = ledgerの `prompt.text`（redact・切詰め済み）の先頭。中断プロンプトは行頭に⚠を表示（独立列は持たない）
- プロジェクト行のcommand列 = `metsuke view period <現在の期間引数> --project <project>`
  （コピー可能な文字列。同一期間のセッション軸絞り込みへ降りる）
- **$/prompt** = セッションの期間内コスト ÷ 期間内prompt数（多prompt型か1件が重い型かの判別）
- **平均ピークctx** = セッション内メインレーンpromptのピークコンテキスト（V4と同一定義）の平均（k tok表示。
  重context型セッションの判別 — 高コストの理由がprompt数か単価かcontextかを他セッションと比較して読む）
- 時間表示は24h超をday換算（例: `2d 23h 25m`）。「期間」列＝wall-clockの規約は変わらない
- 「期間」列はプロンプト軸にも置く（= prompt帰属requestの開始時刻幅。V4の「観測スパン」と同一値・
  V1内の列名はセッション軸と揃えて「期間」）
- $/prompt・平均ピークctx はプロジェクト軸にも置く（project単位の同定義集計）
- **ソート**: ランキング表は **command列を除く全列**がヘッダクリックで昇降順ソート可
  （初期表示はコスト降順。数値は `data-sort` 属性・文字列列（project・費目等）は文字列比較。
  JSは並べ替えのみ）。累積%の値はコスト降順時のまま再計算しない

読み方の注意:

- **分位点（p50/p90/…）はこの画面には出ない** — 分布の形は `metsuke view dist`（V4）で見る
- 「期間」列はwall-clock（開始〜最終リクエスト）。V4の「観測スパン」（プロンプト帰属requestの開始時刻幅・分単位）とは別物
- セッション軸の件数は**自動実行の短命セッションを大量に含む**（設計時サンプルでは大半）。
  insightは「$1以上N件」の閾値表現とし、中央値は出さない（自動実行に潰されるため）
- 未知モデル（cost_usd NULL）の行がある期間は件数を警告表示する

## ユースケース

### UC1 週次レポの裏取り（月曜朝）
- きっかけ: 週次アナリストレポ（push）の「先週$X・projectAが6割」という主張（レポにコマンド併記）
- コマンド: `metsuke view period --week last`
- 見る値: プロジェクト軸でレポと同じ数字を突合 → 該当projectのセッション軸上位
- 判断: レポの帰属は正しいか。意図した集中か
- 行動: 疑わしいセッション行の `metsuke trace <session_id> --html` を実行し背景込みで精査

### UC2 高額プロンプトの位置の確認（V4から）
- きっかけ: V4で領収書ショックの正常性判定をした後、同期間の高額プロンプトの中での位置を確かめたい（遷移表: V4→V1・持ち物=V4と同一の期間引数）
- コマンド: `metsuke view period --week last`（V4に表示された期間引数をそのまま使う）
- 見る値: プロンプト軸ランキングでの当該件の位置・累積%
- 判断: 単発で突出しているか、高額プロンプトが複数並ぶ期間か
- 行動: 複数並ぶなら依頼の仕方（分割・委任・モデル）を
  `metsuke mark start --category <category> --hypothesis "…" --expected "…"`（判断投入CLI・想定）で施策候補に登録

### UC3 請求ズレの帰属（月次の儀式）
- きっかけ: 月次インボイスが体感と合わない（遷移表: 月次請求書 → V1＋V4同時）
- コマンド: `metsuke view period --month 2026-06` と `metsuke view dist --month 2026-06` を並べて開く
- 見る値: V1プロジェクト軸のshare分布（V4では分布の形を同時確認）
- 判断: 集中型（上位3で75%型）か分散型か
- 行動: 集中なら該当プロジェクトの運用だけ見直す。cache費目の異常が見えたらV3へ

### UC4 今日の暴走の帰属
- きっかけ: statuslineで日次ヘッドルームの消化が速い
- コマンド: `metsuke view period --today`
- 見る値: セッション軸で今日の高額セッション・開始時刻・req数
- 判断: 高額が既知の作業セッションに集中しているか、心当たりのないセッションか
- 行動: 心当たりのない行の `metsuke trace <session_id> --html` でfan-out・時間構造を確認してから継続/中断を決める

## 利用実測と撤退

生成時に `hook_event.kind='view_html_generated'`・`payload_json` の `$.view='period'` を記録し、
四半期棚卸しで生成0件ならこのビューを削除する（ADR 0007）。

## 現行プロトからの変更方針（2026-07-18裁定）

1. プロンプト軸insightを分位点→帰属型に差し替え（分位点はV4へ一本化・`metsuke view dist` を併記）。
   空いた枠に**中断プロンプト支出**（設計時サンプルでは少数だが無視できない総額比 — 件数と併せて
   中断対策の優先度を判断する材料）と**帰属不能残余**（同: 0.7%）を収容
2. セッション軸insightの低額な中央値を撤去し「$1以上N件でY%」の閾値集中度に差し替え
3. 「実働」列を「期間」に改名（V4の実働と衝突するため）
4. 期間プリセット（--week/--month/--today/--from--to）と--projectフィルタを本実装のCLI要件とする
   （V2の変化点・V4の帯から「持ち物」を持って降りる受け皿）
5. コマンド併記はID・`--html` を含む完全形で表示する（IDは**8文字プレフィックス** — CLIがプレフィックス解決する。フルUUIDは表を溢れさせるため使わない）
6. セッション軸に $/prompt・平均ピークctx 列を追加（2026-07-18 FB）。列ソート（初期コスト降順）、
   24h超のday換算表示、コストbarの起点揃え（金額桁数の影響を受けない）も同FBで採用。
   期間選択はプロトでも `--days N / --from <start> --to <end> / --today` を実装する
7. プロンプト軸に「期間」列、プロジェクト軸に $/prompt・平均ピークctx 列を追加し、
   ソートを command 除く全列に拡張（2026-07-19 FB）
7. 検討（優先度低）: コストbarのmain/委任2色分割
