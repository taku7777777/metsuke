# V3 cache（キャッシュ健全性ビュー） — write投資の回収診断

- **答える問い**: キャッシュ投資（write費）は**回収されているか** — 破れ（⚡）はいくらの損で、原因はどこか（J5）
- **3秒行**: 「期間内read/write倍率X倍・read節約で未相殺のwrite費 $Y（推定）」
- **次の精査コマンド**: セッション行 `metsuke trace <session_id> --html`（コマンド文字列併記・リンクは生成しない）。
  傾向の疑問 → `metsuke view trend`

## 目的

見えないキャッシュ機構を**ドル建ての診断**に翻訳する。⚡（キャッシュ恒等式の破れ）の
件数ではなく、**write投資が読み出しで回収されているか**という結果で答え、
回収できていないセッションを名指ししてtraceへ渡す。
行動レバー「キャッシュ継続性」（離席前の扱い・再開のまとめ方・切替タイミング）の判断根拠。

## 提供価値

1. **回収の可否が金額で出る**: 期間内read/write倍率と未相殺write費。
   「$Xなら離席運用を変える価値があるか」という実在の判断に直結する
2. **破れの値札**: ⚡由来の推定再構築費を**unknown含む原因別**に金額換算。
   金額はセッション精査の優先順位に使える（unknownは原因別施策には使わず、trace精査か分類器改善へ送る）
3. **効率の悪いセッションの特定**: cache_read率とblended単価でキャッシュ崩壊セッションを
   ハイライトし、trace降下の優先順位を与える
4. **初回write費の要因監視**: 起動コンテキスト量（システムプロンプト/CLAUDE.md/メモリ等・
   セッション初回requestの入力側合計）の週平均をヘッダで提示。
   **初回write費の潜在要因**という位置づけ（実際にwriteされた量はTTL別write列で確認する）

## 画面構成と読み方

- ヘッダ: read/write費・**期間内read/write倍率**・**未相殺write費**・
  起動固定費（tok/session・**セッション数最大のモデルのみ**表示・モデル名とセッション数併記・
  当期と比較期の両方に存在しない場合は比較値を非表示）
- insight: 「⚡再構築費 推定$X（原因別内訳: ttl_expiry $a / compaction $b / … / unknown $u）・
  分類カバレッジY%（件数ベース）」
- 日次⚡件数（**3チャート**・各内を原因別積み上げ・unknownは灰色/凡例「未分類」。2026-07-19 裁定 [06-open-questions Q14]）:
  1. **喪失・放置起因（直前間隔≥5分）** = TTL失効。
     **訂正（2026-07-19・Q15）**: ここを「回避可能（運用hygiene: 離席前の区切り・切替タイミング）」と
     書いていたのは**誤り**。実測で放置1h+の再構築費の約9割は再開後の実作業に償却されており、
     純粋な無駄は小さな比率にとどまった。5〜60分帯も大半は1h writeのTTL内失効＝下記2と
     同じ現象。**この帯の大半は回避不能**として読むこと
  2. **喪失・活動中（直前間隔<5分＝会話キャッシュ全リセット）** = 内容不変なのにキャッシュ一括ミスし全再write。
     深掘り実測で idle TTL/context edit/内容変化はいずれも棄却され、**prompt caching が best-effort
     （TTL内・活動中でも保持保証なし）である挙動と整合＝ユーザー起因ではない**（provider側evictionかCC側かは
     transcriptから区別不可）。V3では「回避可能な放置喪失」と分けて提示する
  3. **増加（参考・損失ではない）** = gap>0。別経路のキャッシュ読み戻し。約8割は直前ターンのoutput再読の
     会計上の偽陽性。増加側は再構築費総額の約1%と金額影響が軽微なため補正は見送り、件数・causeの
     解釈時のみ留意する
  （符号分離で「損失」系unknownは約半分になった）
- 日次費目バランス（read vs write棒＋w1h選択率折れ線）
- セッション上位表（write費順）:
  sid / project / ⚡上位原因 / write 5m費・1h費($) / **cache_read率** / 主モデル / blended単価 / read費($) / command（`metsuke trace <session_id> --html`）
  — **cache_read率<90% かつ セッション総額cost_usd≥$1 の行をハイライト**（ハイライトは非混合行に限定）

指標の定義（本実装時にMETRICS.mdへ正式定義を置く）:

- **期間内read/write倍率** = 期間内 `SUM(cache_read_tok) ÷ SUM(cache_w5m_tok+cache_w1h_tok)`。
  **因果対応なし**（期間開始前に作られたキャッシュのread・期間末のwriteの将来readを含まない打切りバイアスあり）
- **write費** = `cache_w5m_tok×w5m単価 + cache_w1h_tok×w1h単価`、
  **read節約額** = `cache_read_tok×(通常input単価−cache_read単価)`、
  **未相殺write費（推定）** = `max(0, write費−read節約額)`。
  「投資回収」と同義にはしない（write費全額を投資と見なす近似。キャッシュ化の追加費用ベース
  `(write単価−input単価)` での精密化はMETRICS正式化時に裁定）。
  単価は `v_request_cost` と同じSCD2解決・batch/fast/geo係数適用。write→read非対応の期間集計近似である旨を脚注
- **未知価格の共通契約**: 価格を使う全ドル指標（read/write費・未相殺write費・⚡再構築費・blended単価）は、
  対象requestに価格不明（cost_usd NULL）があれば「価格カバレッジ不足」を警告し件数を併記する
- **⚡再構築費** = 各破れ行のcurrent requestを `v_request_cost` に結合し、その行の
  `cache_w5m_tok/cache_w1h_tok`×各write単価で評価（METRICS §4準拠。`gap` は検出専用で費用式には使わない）
- **cache_read率** = `SUM(cache_read_tok) ÷ SUM(input_tok+cache_read_tok+cache_w5m_tok+cache_w1h_tok)`
  （入力側生トークン分母・セッション粒度）。閾値90%は全モデル共通の運用閾値
  （根拠: 設計時サンプルで率<90%群のblended単価中央値が≥90%群の約4倍となり、
  90%を境に段差状に悪化した）
- **主モデル/混合** = 入力側生トークンが最大のモデルを主モデルとし、主モデル比率<90%を「混合」とする。
  blended単価の比較・ハイライトは非混合行に限定
- **blended単価（$/Mtok）** = `SUM(cost_usd) ÷ (SUM(input_tok)+SUM(COALESCE(output_tok,0))+SUM(cache_read_tok)+SUM(cache_w5m_tok)+SUM(cache_w1h_tok)) × 1e6`
  （中断output未計上のため過小方向の脚注を付す）
- **起動固定費** = v_context_overhead の `startup_context_tok`（tok/session・初回requestの入力側合計）

読み方の注意:

- **blended単価はモデル間の単価比較には使えない**。キャッシュヒット率で変動する効率指標
  （設計時サンプルでは同一モデル内でcache_read率<90%群が≥90%群の約4倍）。
  分母は生トークン合計を維持する（価格ウェイト式にすると分散が消え診断力が失われる — 裁定済み）
- unknownは**原因分類のみ未確定**（破れの検出と再構築費の金額は診断済み）。金額はセッション精査の
  優先順位には使い、原因別施策の優先順位には使わない
- read/write費の費目分解は `v_request_cost` と同じ係数を適用し、費目合計＝総額の入力側部分と一致させる
- 中央値系の数値は自動実行セッションを含む。「対話のみ」併記値で読む（判別は台帳共通のproject名分類）

## ユースケース

### UC1 領収書のcache_w違和感の裏取り
- きっかけ: 昼休み後の再開でcache_w1hが太い領収書を見た（遷移表: 領収書→V3・持ち物=当日）
- コマンド: `metsuke view cache --today`
- 見る値: w5m/w1h内訳・ttl_expiryの件数と再構築費・該当セッション行・
  insightの「同一project・直近4週のttl_expiry回数」
- 判断: ttl_expiry型（放置後の重い再開）が**当日を含む直近4週で同一projectに2回以上**あるか
- 行動: 反復なら再開をまとめる/離席前にhandoffする等の運用変更を `metsuke mark`（判断投入CLI）で施策化

### UC2 V2費目異変からの降下
- きっかけ: V2費目タブでwrite費が急増した週を見つけた（遷移表: V2費目タブ→V3・持ち物=期間）
- コマンド: `metsuke view cache --from <start> --to <end>`
- 見る値: セッション上位表のハイライト行（read率<90%∧$1以上）と⚡上位原因
- 判断: cause別に対応レバーを選ぶ（ttl_expiry→放置・再開運用 / model_switch・config_change→切替タイミング /
  compaction→コンテキスト運用 / interruption→中断後の再開パターンをtraceで確認 /
  unknown→原因別施策は選ばずtrace精査または分類器改善へ）
- 行動: 最上位の1本を `metsuke trace <session_id> --html` で精査して原因を確定

### UC3 月次の習慣ベンチ
- きっかけ: 月次請求突合の儀式（遷移表: 月次請求書→V3）
- コマンド: `metsuke view cache --month <YYYY-MM>`
- 見る値: 期間内read/write倍率・未相殺write費・原因別再構築費の前月比・
  起動固定費（同一モデルで比較可能な場合のみ）
  （比較規則: `--month` は前月比・`--week`/既定は前週比。ラベルも動的に切替）
- 判断: キャッシュ経済が悪化傾向にないか。起動固定費が育っていないか（同一モデル比較可能時のみ・補助）
- 行動: 悪化していれば `metsuke mark` で施策候補化（CLAUDE.mdダイエット・セッション運用の見直し）

## 利用実測と撤退

生成時に `hook_event.kind='view_html_generated'`・`payload_json` の `$.view='cache'` を記録し、
四半期棚卸しで生成0件ならこのビューを削除する（ADR 0007）。

## 現行プロトからの変更方針（2026-07-18裁定）

1. **問いを再定義**: 「キャッシュは効いているか（⚡件数）」→「write投資は回収されているか（金額）」。
   台帳（07-views.md）の行も更新済み
2. ヘッドラインを結果指標へ: 期間内read/write倍率＋未相殺write費を1〜2番目に。
   ⚡は「原因別内訳（unknown含む）・分類カバレッジY%」の表記に降格
3. insightをドル建て化: ⚡再構築費の原因別金額（METRICS §4の式・unknownも金額表示）
4. セッション表にcache_read率列・主モデル列を追加し、「率<90%∧総額$1以上」ハイライトを導入
   （**未決事項「$/Mtok外れ値閾値」はこれで裁定・クローズ**）
5. $/Mtok列は「blended単価」に改名し補助情報へ。脚注定型文を必ず併記
   （**未決事項「実効トークン定義」は生トークン分母維持で裁定・クローズ**。
   価格ウェイト式は「請求異常検知」の問いが立った時のみ再検討 — 台帳の却下表）
6. ヘッダに起動固定費（tok/session・週平均・対象セッション数併記）を追加
7. unknown分類器の改善は引き続き台帳課題（docs/06-open-questions.md）— ビュー側はカバレッジ表記で自立
8. CLI要件: `--today` / `--from <start> --to <end>` / `--month` を受ける（上流からの「持ち物」の受け皿）
9. insightに「同一project・直近4週のttl_expiry回数」を追加（UC1の反復判定の根拠値）
10. ⚡件数チャートを **gap符号で喪失(−)/増加(+) に分離**（2026-07-19 FB・[06-open-questions Q14]）。
    符号の意味: 喪失(−)=実read(n+1)が read(n)+write(n) を下回り**キャッシュを失った→再構築費が立つ**
    （TTL失効・model_switch・**compaction もここ**: クリーン事例で read 224k→20k と確認）。
    増加(+)=上回り**別経路で作られたキャッシュを読んだ・喪失ではない**（正体は未特定）。
    unknown過多の要因は「cause判定の時間窓が効いていなかったスコープバグ（2026-07-19 修正済み・
    修正後カバレッジが約4割から約3割へ低下し、その後5m-TTL層で数ポイント回復）＋計装は途中から
    （開始日はハードコードせずビュー側が動的に算出する）」
    ＋「検出モデルが喪失と増加を同一視」＋「TTL規則が1hのみ」。符号分離と5m-TTL層は実装済み。
    `compaction` / `config_change` はhook記録開始前には判定不能であるため、表示期間の⚡に占める
    計装前件数・割合を動的な注記として表示する。`+output(n)` 補正は金額影響が軽微なため見送った
11. **区分軸を「回避可能性」へ切り直す（2026-07-19 提案・未実装・根拠 [06-open-questions Q15]）**。
    現在の3区分（放置/活動中/増加）は**現象の分類であって行動の分類ではない**。実運用ループを
    1周回した結果、「放置＝回避可能」という現行の読み方が誤りだと実測で判明した。行動を決める
    ための区分は次の3層:
    - **不可避（TTL物理・償却済み）**: 1h+放置からの再開で、再開後に実作業が続いたもの（費用の大半）。
      提示はするが行動対象にしない
    - **provider起因（best-effort eviction）**: 活動中<5分 ＋ 1h TTL内での失効（大きな費用群）。
      ユーザー起因ではない。Q14の結論をこの層のラベルとして表示する
    - **回避可能（全再構築費の1桁%）**: 再開したが後続作業がほぼ無いもの（≤2req）＋ 5m書込のみで
      5〜60分放置により失効したもの。**ここだけが `metsuke mark` の対象**
    実装時の注意: 償却判定は「再開直後の後続リクエスト数」で行い、窓は**2時間を既定**とする
    （30分/2h/6h で符号は一致・30分は作業再開の遅れを非償却と誤認しやすい）。
    層の切り分けは開始基準・終了基準のどちらでも所属が変わらないことを確認済み（Q15）
