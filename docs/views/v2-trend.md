# V2 trend（推移ビュー） — 変化の検知と施策検証の場

- **答える問い**: 週次の費用構成と行動指標は**変わったか** — その変化はどの軸（費目/モデル/プロジェクト）・どの施策期間に対応するか（J3）
- **3秒行**: 「今週 $X（先週同時点比 ±Y%）／最大{増加|減少}寄与: `<project>` ±$Z」（符号は実測に従う）
- **次の精査コマンド**: 変化点の週 → `metsuke view period --from <start> --to <end>`。
  費目タブのcache異変 → `metsuke view cache --from <start> --to <end>`

## 目的

「変わったか」に答える唯一の時間軸面。日次×3軸の積み上げで**変化点の時期と帰属**を
同時に特定し、施策マーカーで**前後検証（J3のループ）**を成立させる。
週次最適化サイクルの月曜の儀式のアンカー。

## 提供価値

1. **変化点検知**: 日次積み上げ＋7日移動平均線で「いつから変わったか」が読める
2. **変化の帰属**: タブ切替（費目→モデル→プロジェクト）と週次サマリの最大寄与で「何が動かしたか」を特定
3. **施策前後検証**: 施策期間の帯・交絡イベントの縦線を時間軸に刻み、適用日の暗記に依存しない
   before/after 比較を可能にする。`expected_effect`（自由記述）と画面指標の対応は**自動化しない** —
   全指標を表示し、人間が照合する（機械可読なmetric/期待方向の記録は将来のmarker拡張課題）
4. **傾向の3関心**: 週次挙動サマリで自律性（委任費率）・コンテキスト量（≥200k費率）・
   高性能モデル使用（fable+opus費率）の推移が1表で追える
5. **分布形の推移**: 対話のみのプロンプトcost分位点（p50/p90/p95）とピークctx分位点（p50/p90）の
   週次系列で、「1依頼の相場」「コンテキスト習慣」のドリフトを検知する
   （設計時サンプルでは短期間にcost中央値が1桁以上、context中央値も約1桁増えた。
   総量だけでは見えない行動変化だった）。
   断面の形はV4・推移はV2の分担（定義共有）
6. **行動イベントの推移**: gap>1h再開・送信後破棄（中断）・compactionの週次件数で、
   キャッシュ断絶やワークフロー変化の兆候を件数で追う（原因別⚡の断面診断はV3の領分）

## 画面構成と読み方

4段構成（**粒度タブ 日次/週次/月次 は①〜④すべてに適用** — 粒度で見えるものが変わる:
日=スパイクの帰属・週=意思決定の主粒度・月=ポリシー/請求の変化）:

- ①**総量**: 3タブ（費目/モデル/プロジェクト）× 積み上げ棒＋**7日移動平均線（日次のみ）**＋**施策マーカー**（下記）
- ②**分布形の推移**: 折れ線3枚 — 対話のみの cost p50/p90/p95＋**平均**、ピークctx p50/p90、
  **実行時間（観測スパン）p50/p90/p95**（分単位。V4と同一定義 = prompt帰属requestの開始時刻幅・
  処理時間ではなく人間待ち等を含む代理指標）。
  **日次はn≥20の日のみ点を描画**（少数日の分位点は不安定 — 欠けた日は線を途切れさせ注記）
- ③**行動イベントの推移**: gap>1h再開・中断プロンプト・compaction の件数系列（選択粒度で集計）
- ④**挙動サマリ（グラフ＋表）**: 表の上にグラフ3枚（いずれも選択粒度に追従）:
  費率3系列（委任費率・fable+opus費率・≥200k費率 — %軸共通）／
  件数2系列（対話n・高reqプロンプト数）／ blended$(対話)
- ④の**挙動サマリ表**（選択粒度の行単位）:
  期間 / 合計 / 前期比（同経過日数） / 利用日数 / 最大寄与project / **対話n** / 委任費率 /
  fable+opus費率 / ≥200k費率 / **blended$(対話)** / **高reqプロンプト数**。
  比較列は粒度に応じ前日比（当日は進行中注記）/前週比（WTD）/前月比（同経過日数）
- 月次はデータ蓄積が1ヶ月未満の間は1行/1点表示（無効化はしない — 蓄積とともに意味を持つ）
- insight行: 3秒行と同一テンプレート（週次基準・粒度によらず固定）

指標の定義:

- **観測日**: ledgerの計測開始日（`MIN(request.ts)` の暦日）以後の暦日。それ以後の `v_daily` 欠行は
  $0（利用なし）とみなす（取込停止との区別は `metsuke doctor` / v_health の担当 — この限界を注記で明示）
- **前週比（同経過日数・week-to-date)**: 当週の経過日数をDとし、**前週の先頭D日がすべて観測日**の
  場合のみ、同経過日数の累計同士で比較して表示する（計測開始週などは非表示）。減少は負値で表示
- **利用日数n/7**: `v_daily` に行が存在する日数。計測完全性ではなく**利用のあった日数**（表示情報であり
  比較可否の判定には使わない）
- **7日移動平均**: ローカル日付のカレンダースパインを生成し、観測日の利用なし日を$0補完した上で
  当日を含む7暦日平均（計測開始日前は欠損扱いで線を引かない）
- **最大寄与project**: 同経過日数における project別差分（当週−前週・新規/消滅projectは$0補完）のうち、
  総差分と同符号で絶対額最大のproject。減少週は「最大減少寄与 `<project>` -$Z」と表示
- **委任費率** = `SUM(cost_usd WHERE agent_id IS NOT NULL) / SUM(cost_usd)`（週内の全request）
- **fable+opus費率** = `model LIKE '%fable%' OR model LIKE '%opus%'` の `SUM(cost_usd) / SUM(cost_usd)`
  （未知価格 `cost_usd NULL` は分子分母から落ちるため、該当件数がある週は警告注記）
- **≥200k費率** = メインレーンのプロンプト別ピークコンテキスト（V4と同一定義）が≥200kの
  プロンプトのコスト ÷ 帰属済みプロンプト総コスト
- **cost/ctx分位点（対話のみ）** = 週内の対話プロンプト（台帳共通の自動実行分類で除外）の
  コスト分位点、およびピークコンテキスト（V4と同一定義）の分位点
- **gap>1h再開** = メインレーン（agent_id IS NULL）の同一session内連続request間隔が3600秒超の回数
  （ttl_expiry原因件数より検出範囲が広い上流指標）
- **中断プロンプト** = 週内requestで `MAX(is_interrupted)=1` となったprompt数
- **compaction** = v_cache_identity の cause='compaction' 件数
- **高reqプロンプト数** = 期間内の対話プロンプトで、プロンプト帰属のリクエスト数（サブエージェント分含む）が
  **20以上**の件数（メインレーン長ループの推移）。閾値20は設計時裁定 —
  実測でcost上位10%のrequest数が残り90%より1桁多かったことから長ループ型を拾う値として採用。
  分布が変われば要調整
- **blended$(対話)** = 週内対話requestの `SUM(cost_usd) ÷ SUM(トークン計・outputはCOALESCE 0) × 1e6`
- **施策マーカー**: markerは `ts_start`〜`ts_end` の期間帯（`ts_end` NULLは表示期間末まで）・
  titleは**定型値のみ**（category・verdict）。regime_eventは `ts` の縦線・titleはkindのみ
  （hypothesis/detail等の自由記述はtitleに出さない — PII面「数値のみ」を維持）
- **before/after窓**: after窓は `ts_start` から7暦日（施策が途中終了した場合は `min(ts_end, ts_start+7日)`）。
  経過が7日未満の間は「暫定（n日経過）」と表示し verdict 記録は推奨しない。
  窓が他markerと重なる場合は交絡として警告表示

読み方の注意:

- 日次棒の直読は**ワークロード混合に注意**（異種ワークロードが混ざるとモデル構成が大きく変わる）。
  傾向は移動平均線で読む
- 管理図・同曜日比・複数窓の移動平均は**不採用**（管理図のみ8週蓄積後に再検討 — 台帳の却下表）
- 予算線は置かない（ペーシングはJ2=statusline担当。V2は事後の変化検知に徹する）

## 主データ源

総額・移動平均は `v_daily`。軸別日次と各費率は `v_request_cost`＋`session`
（≥200k費率はprompt単位へ集約）。施策注記は `marker`・`regime_event`。

## ユースケース

### UC1 月曜の前週比の帰属
- きっかけ: 週次アナリストレポの「先週比+48%」主張（レポにコマンド併記。遷移表: 週次レポ→V2）
- コマンド: `metsuke view trend`
- 見る値: insightの同時点比・週次サマリの最大寄与project・3傾向列
- 判断: 意図した作業由来（本業リポの集中）か、バッチ実験由来か、単価劣化（V3の領分）か
- 行動: 意図内なら承認して終わり。意図外なら該当週を `metsuke view period --from <start> --to <end>` で帰属

### UC2 施策の前後検証
- きっかけ: 週次レポに未判定markerの検証コマンドが併記された（遷移表: 週次レポ→V2）
- コマンド: `metsuke view trend`
- 見る値: 施策帯の前後7日での `expected_effect` 対応指標（例: cache_read/write構成・委任費率）・
  交絡の縦線（regime_event）の有無
- 判断: `expected_effect` に対応する指標（人間が照合）が期待方向に動いたか。交絡はないか
- 行動: `metsuke mark verdict <marker_id> win|loss|inconclusive`（判断投入CLI・想定）で記録。
  効かなければ `metsuke mark start --category <category> --hypothesis "…" --expected "…"` で次の施策を1本立てる

### UC3 費目異変からのV3降下
- きっかけ: 費目タブでcache_w帯が太った週を見つけた（遷移表: V2費目タブ→V3・持ち物=期間）
- コマンド: `metsuke view cache --from <start> --to <end>`
- 見る値: →V3でwrite費上位セッションと⚡原因
- 判断: 反復パターンか単発か
- 行動: 最上位セッションの `metsuke trace <session_id> --html` で精査

## 利用実測と撤退

生成時に `hook_event.kind='view_html_generated'`・`payload_json` の `$.view='trend'` を記録し、
四半期棚卸しで生成0件ならこのビューを削除する（ADR 0007）。

## 現行プロトからの変更方針（2026-07-18裁定）

1. 前週比を同経過日数比較（week-to-date）に変更し、利用日数n/7列を追加。
   比較対象側の観測が不足する週の比は非表示
2. 直近4週表を「週次挙動サマリ」へ拡張: 最大寄与project＋委任費率・fable+opus費率・≥200k費率
   （ユーザーの3関心を新ビューを増やさず充足する）
3. 施策マーカー注記レイヤ（marker=期間帯＋verdict、regime_event=縦線）をチャートに追加
4. 7日移動平均線を1本追加（カレンダースパイン・$0補完）。
   **未決事項「移動平均等の軸追加」はこれで裁定・クローズ**
   （管理図=8週後再検討・同曜日比=不採用・複数窓=不採用）
5. insight行を新設（4ビュー中V2だけ無い状態を解消・対称性）
6. パネル下に `metsuke view period --from <start> --to <end>` のコマンド併記（降下の可視化）
7. 予算参考線は置かない（J2との役割分担を明確化）
8. 4段構成へ再編（2026-07-19 FB）: 分布形の推移（対話のみcost/ctx分位点）・
   行動イベント推移（gap>1h/中断/compaction）を新設し、挙動サマリに対話n・blended$(対話)・
   高reqプロンプト数を追加。ビューは分割しない（問いは同じ「変わったか」— 粒度はパラメータ。
   撤退基準の管理対象を増やさない）
9. 粒度タブ（日次/週次/月次）を①〜④全セクションに適用（2026-07-19 FB第2弾）。
   ガード: 日次分位点はn≥20の日のみ・月次は蓄積1ヶ月未満の間は1行/1点表示
