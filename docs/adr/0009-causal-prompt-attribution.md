# ADR 0009: agentリクエストの因果プロンプト帰属（Q13の決着）

日付: 2026-07-18 / 状態: 採択（ユーザー承認済み — 因果帰属へ切替）

## 決定

`request.agent_id IS NOT NULL` の行の `prompt_id` を、時間帯ベースの観測値ではなく
**因果帰属**（`agent.parent_tool_use_id → tool_call → 親request.prompt_id` = spawn時の
アクティブプロンプト）で確定する。プロンプト集計の意味論は
**「この指示が引き起こした総コスト」** に統一される。

- 取込規則8として [02 §2](../02-data-model.md) に追加し、golden fixture でテスト固定。
- 実装は ingest の derive 段（毎run・冪等）。ネストagentは spawn 連鎖を親から順に
  解決する（決定的）。親 tool_call へ辿れない agent のみ観測値を温存（実測では
  agent系コストの1%未満）。
- `tool_call.prompt_id` は「所属requestと常時同期」に変更（従来はNULL埋めのみ）。
- PARSER_VERSION 4→5。`metsuke rebuild` で全履歴に遡及。mainレーンの帰属
  （lineage_state = 直近user行のpromptId）は不変。

## 根拠

1. **一次情報の意味論はmetsukeの問いに合わない**: transcript の `promptId` は user 行のみが
   持ち、課金実体の assistant 行には無い — request→prompt 帰属はもともと metsuke 定義の
   二次情報である。かつ agentレーン user 行の `promptId` は「記録時点の main アクティブ
   プロンプト」を刻む（実測: 走行中agentのuser行が起動元プロンプト→到着した
   task-notification プロンプトへ切り替わる）。このため走行中agentのコストが後続の
   通知ラリーへスライスされることを実運用traceで確認した。
2. **誤帰属の規模が意思決定を歪める**（数万request規模の実運用台帳で定量）:
   - agent系コストの**約4分の1、総コストの約5%**が時間帯帰属と因果帰属で行き先が異なった。
   - プロンプト別ランキングで数百位から1桁順位への逆転が発生
     — V1 period ビューの外れ値検出が現行帰属ではミスリーディングになる。
   - 因果帰属で解決できない残余は1%未満のみ。
3. **情報の損失なし**: 一次情報は無加工アーカイブ（ADR 0001）＋決定的 rebuild で、
   時間帯ベースの帰属はいつでも再導出できる（必要になれば PARSER_VERSION 繰上げで
   観測値列を追加可能）。
4. **波及が自動で正しくなる**: `v_prompt_cost` は `request.prompt_id` の GROUP BY、
   trace のプロンプトグルーピングも同列参照のため、V1 period ビュー・週次アナリスト・
   trace のレーンスライス断片（6-10 で観測）はこの一箇所の規則変更で全て追随する。

## 条件と検証

- golden fixture に「プロンプト境界をまたいで走行する非同期agent」ケースを追加し、
  全agentリクエストが spawn プロンプトへ帰属することを期待値で固定。
- 増分ingest == rebuild の `_table_hash` 一致を維持（derive は毎run冪等・全行対象）。
- 実データ検証では通知プロンプトへのagent誤帰属がspawn元プロンプトへ移ることと、
  traceのレーン断片が解消されることを確認。
- 暗黙ルール化の防止: 本ADR・02 §2 規則8・SCHEMA.md の prompt_id 意味論注記の3点で
  「一次観測値と意図的に乖離する」ことを明文化する。
- 既知の理論的限界（独立レビューで合成データにより確認・実データでは発生し得ない）:
  spawn 連鎖が**循環**する場合（k∤16 の循環長で帰属が run ごとに回転）と**深さ17超**の
  チェーンでは、反復上限16により増分とrebuildの最終状態が一致しない。実際の spawn は
  時間的 DAG のため循環は因果的に発生せず、深さも1桁 — 観測されたら上限見直し。
  fixture はネスト spawn を「観測値が誤っている側の request」から発行させ、固定点ループと
  tool_call 上書き同期の両方が正解に必須になる形で退行を検出する（単一パス化で golden が落ちる）。

## 棄却した代替案

- **観測値列（observed_prompt_id）の並存**: 一次情報＋決定的rebuildから再導出可能な値の
  重複保持になる。監査要求が実際に生じた時点で追加すれば足りる。
- **ビュー側での帰属切替（別列で両立）**: 全ビュー・レポ・trace に分岐が増え、
  「プロンプトのコスト」が文脈依存になる — 1画面=1問いの原則（ADR 0007）に反する。
- **現状維持＋trace表示のみ改善**: 総コスト5.3%の誤帰属とランキング逆転が残り、
  判断支援ビュー（07-views V1）の根拠数字が信用できない。
