# ADR 0007: 判断支援ビュー — オンデマンド生成HTML群の枠

日付: 2026-07-18 / 更新: 2026-07-21 / 状態: 採択・**V1〜V4 実装済み**
（対象確定FBを反映 — V1は3軸・降下動線は当面コマンド併記。2026-07-19 に描画モデルを裁定し
`src/metsuke/viewgen/` として本実装。プロト由来の数値・描画はゴールデン照合で同一性を確認済み）

> 2026-07-21: ビューの問い・指標所有・query定義は本ADRを維持する。自己完結HTMLを通常の
> 提供面とする決定と、コマンド併記だけでtraceへ降りる方式は[ADR 0011](0011-local-dashboard.md)が
> 部分置換する。動的dashboardは[08-dashboard](../08-dashboard.md)に設計し、実装までは現行CLI/HTMLを使う。

## 決定（案）

「判断支援ビュー」という枠を定義する: 良し悪しの断定はシステムが行わず、**判断材料を
気づきやすい形で提示する** pull 型のオンデマンド生成HTML群。ADR 0003 の禁止対象は
「常設・監視画面」であり続け、本枠はその例外ではなく **trace（ADR 0006）と同じ様式**
— オンデマンド生成・自己完結・常設サーバなし・外部送信なし・使い捨て — の水平展開である。

- **対象管理**: [07-views.md（ビュー台帳）](../07-views.md) が唯一の管理場所。
  「答える問い・見るきっかけ・次の行動」を1行で書けない画面は作らない（1画面=1問い）。
- **データ境界**: ビューは **ledger の数値と redact 済みテキストのみ**を読む。
  transcript 本文の読み出しは trace だけの特権に留める（ビューの PII 面を trace より
  一段軽く保つ。ledger 側 prompt.text は redact→切詰め済み — ADR 0006 §6）。
- **生成物**: `~/.metsuke/views/<view>.html`（mkdir 0700・ファイル0600・上書き・
  フッタに redaction_version/parser_version 刻印・旧版 purge — trace と同じ規範）。
- **利用実測**: 生成のたびに spool 経由で hook_event（kind='view_html_generated'、
  payload にビュー名）へ記録。Stage 8実装後は`dashboard_view_opened`を同じ分母へ加え、
  **使われないビューは削除**する。自動preview等は分母へ含めない。
- **セキュリティ**: ADR 0006 §1〜8 を全て継承（CSP・JSON blob 1箇所・`<`全置換・
  textContent/setAttribute のみ・パス安全・外部共有/Artifact アップロード禁止）。
  リンクは当面**一切生成しない**（外部リンク全面禁止のまま）。trace への降下動線は
  `metsuke explain/trace <id> --html` の**コマンド文字列併記**で代替し、`~/.metsuke/` 配下
  相対リンクのみ許可する規範追記は、ビュー群の構成が固まった段階で再検討する
  （2026-07-18 FB: 後回しと決定）。
- **共通基盤**: ビューが2枚を超えた時点で共通スケルトン（テンプレ分裂禁止の拡張）を
  導入する。テンプレ1枚・無ビルド・集計/レイアウトは Python 側・JSは操作系のみ。
- **描画モデル（2026-07-19 裁定）**: 当初「trace と同じ様式＝JSON blob 1箇所」と
  「レイアウトは Python 側」を併記していたが、この2つは両立しない（trace は JSON blob を
  JS が `textContent` で DOM 構築する方式）。ビューは**後者**を採り、trace の
  「JSON blob 1箇所」は継承しない。代わりに XSS 耐性を**規律ではなく構造**で担保する:
  ビルダは生のマークアップを一切書かず、`viewgen/render.py` のプリミティブに
  データ構造（`Cell`/`Column` 等）を渡すだけとし、**マークアップを出力してよい唯一の
  モジュールを render.py に限定**してエスケープをそこへ集中させる。この不変条件は
  「render.py 以外の `viewgen/` 配下にタグ様文字列リテラルと `Html(` 呼び出しが
  存在しないこと」を検査するテストで機械的に強制する。差し込み点はテンプレのマーカー
  （`__VIEW_TITLE__` 等）とし、各マーカーがちょうど1回出現することを検査する。
  trace 側（transcript 本文を扱う＝PII面が重い）は従来どおり JSON blob 方式を維持する。
- **CLI**: `metsuke view <name> [--days N | --today | --week [last] | --month [last|YYYY-MM] |
  --from A --to B] [--project SLUG] [--as-of YYYY-MM-DD] [--open]`。
  期間のアンカーは実時刻の今日、`--as-of` で上書きできる（過去時点の再現・回帰検証用）。
  台帳の数値がアンカーより古い場合はフッタに「データ最終: YYYY-MM-DD」を刻んで
  鮮度のズレを可視化する（無言で窓をずらさない）。

## 根拠

- 気づきの階段の中段が空白: statusline（常時）→ 領収書（瞬間）→ 週次アナリスト
  （push・仮説）→ **【空白: pull・精査】** → trace（深掘り）→ raw。高コスト＝問題とは
  限らず、背景・意図込みの妥当性精査は人間の仕事（00-vision J2「止めるのは人間」/
  J7「読んで承認するだけ」）。ビューはその判断を支援する提示に徹する。
- 実データ試算では費用が上位プロジェクトへ集中していた。
  session.project は全セッションで充填済みで、v_daily / v_prompt_cost /
  v_cache_identity が既存 — 台帳の V1/V2/V3 は**新規データ取得なし**で成立する。
- 進め方はユーザー合意済み: ①台帳で対象精査 → ②優先度順に実データプロト→FBで
  作り込み → ③実装（テスト・契約・生成記録） → ④全体調整・共通化。

## 条件と検証

- プロトは初日から実データ（ledger）で作る。プロトと実装の境界は「データの真偽」では
  なく「作り込みの質」（テスト・redaction 契約・CLI 統合・生成記録）。
- **移植の受入基準（2026-07-19 実施）**: プロト出力から機械抽出した基準線
  （表のセル全体のSHA256・金額出現回数マップ・SVGのtitleハッシュ・insight本文）と
  本実装の出力を突合し、4枚すべてで一致すること。さらにヘッドレスChrome（1440px）で
  描画してピクセル比較し、フッタ刻印を除く全領域が一致することと、
  `scrollWidth == clientWidth`（body横スクロール無し）を実測すること。
  セキュリティテストは変異注入（エスケープ無効化・フィルタ無効化）で
  空振りしていないことを確認する。
- 撤退基準: 四半期棚卸しで`view_html_generated + dashboard_view_opened`が0件のビューは削除。
  枠ごと使われなければ本 ADR を廃止し 07-views.md を削除する。

## 棄却した代替案

- **Datasette 据え置きで足りる**: ad-hoc SQL はあるが「意図した並び・外れ値の強調・
  trace への降下動線」がなく、気づき支援には不足。ビューは Datasette を置換しない（共存）。
- **運用の重い常設サーバ / 既製BI**: ADR 0003 の棄却理由のまま変更なし。軽量loopback dashboardは
  CLI/HTMLの実運用摩擦を根拠にADR 0011で後から採用した。
- **週次アナリストレポートへの一本化**: push 型のみでは「疑問が湧いた瞬間の pull」を
  満たせない。レポートは根拠リンクとしてビューへ接続する（対にする）。
