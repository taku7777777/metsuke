# ADR 0003: ダッシュボードを作らず、注意の3層（statusline / hooks / 週次AI）に投資する

日付: 2026-07-17 / 状態: **部分的に置換**（2026-07-21・[ADR 0011](0011-local-dashboard.md)）

> 本ADRは当時の判断記録として残す。Grafana等の汎用監視基盤、外部公開、リアルタイム監視、
> 運用の重いサーバ群を採用しない判断は有効。一方、ローカルWebプロセスまで一律に禁止する部分は、
> CLI引数の発見性と画面遷移の実運用上の摩擦を受け、ADR 0011が置換した。

## 決定

常設のダッシュボードサーバ（Grafana/Loki/Prometheus/Tempo等）を持たない。
人間向けインターフェースは次の3層に限定する:

1. **ambient**: statusline 1行（常時視界・ゼロコストのちら見・鮮度⚠兼用）
2. **just-in-time**: hooks（再開時cold-cache警告・API換算の予算警告・暴走ガード・Stop領収書）
3. **deliberate**: `metsuke` CLI（Tree/Waterfall TUI）＋Datasette（呼んだ時だけ）＋週次AIレポート

## 根拠

- 目的関数は行動変容（J8）。「見に行く画面」は行動の瞬間から遠く、気づき→行動の距離が
  減衰器になる。行動が決まる場所（プロンプト入力欄の前）にフィードバックを置くのが最短。
- ダッシュボード保守（定義生成器・パネルドリフト・サーバ運用）は単一保守者の予算を食う
  （旧リポジトリで実証済み）。
- AIアナリストにとってもピクセルは劣った口。SQL＋テキストが最良。

## 条件と検証

- ナッジは発火と「10分以内に行動が変わったか」（conversion）を計測し、効かない介入を特定する。
  ただし自動ミュート権限は与えない（帰属ヒューリスティックのバグで有効なナッジが消える事故を防ぐ）。
- 警告疲労対策: ルール別cooldown・日次上限・必ず代替行動＋金額を添える。
- 深掘り需要が CLI/Datasette で満たせない事態が続く場合のみ、ローカルWebビュー（読み取り専用・
  ledger.db 直読み）を追加検討する（サーバ常設はしない）。
  → 2026-07-18 この条件が成立し、**サーバレスのオンデマンドHTML生成**として発動
  （[ADR 0006](0006-html-trace-view.md)）。常設禁止の決定は不変。

## 棄却した代替案

- Grafana維持: 上記の理由に加え、旧構成との決別（ゼロベース）を優先。
- Langfuse/Phoenix等の既製UI: データモデル不一致・6コンポーネント級の運用・施策検証機能の不在。
