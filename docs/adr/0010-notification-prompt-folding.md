# ADR 0010: task-notification プロンプトの起点への畳み込み（ADR 0009 の完結）

日付: 2026-07-18 / 状態: 採択（ユーザー承認済み — 畳み込みを選択）

## 決定

task-notification（バックグラウンドタスク完了時にハーネスが注入する擬似プロンプト）に
帰属している request を、**タスクの起点プロンプト**へ再帰属する。起点の解決は
2経路の優先順（いずれも ADR 0009 と同じ機械的因果連鎖）:

1. **第一経路**: 通知本文の `<tool-use-id>` → `tool_call` 直接参照 → 親request.prompt_id。
   タスクを起動したtool呼び出しそのものを指す最も直接的なエッジで、agent表に行が無い
   背景タスク（MCP/Codex・workflow等）もカバーする。
2. **フォールバック**: `<task-id>`（hex）→ agent → parent_tool_use_id → tool_call 連鎖
   （tool-use-id が本文に無い agent 型通知用）。

- 識別は**本文先頭アンカー**: prompt.text が `<task-notification>`（または
  SYSTEM NOTIFICATION 前置き＋`<task-notification>`）で始まる場合のみ。本文中に
  通知を引用しただけの人間プロンプトは畳み込まない。
- ADR 0009 の derive 固定点ループに規則として統合（通知畳み込み → agent因果帰属 →
  tool_call同期が相互に連鎖するため）。通知ラリー中に起動された次のagent波も
  推移的に人間の起点へ到達する。
- 連鎖不能（task-id が台帳の agent に不在・spawn連鎖欠損）は現状維持で温存。
  複数 task-id を含む通知は先頭の task-id で決定（決定的タイブレーク）。
- **prompt 行自体は削除しない**: text・時刻は残り、request が0件になるため
  v_prompt_cost には現れなくなる。trace のラリー一覧は「request 0件のプロンプトを
  表示しない」既存挙動により通知の ⚑ エントリも非表示になる（agentの最終報告は
  起点ビューの agent レーン最終 request / Agent ツール詳細から参照可能。⚑ ナビの
  復活が欲しくなったら表示層FBとして起票）。
- PARSER_VERSION 5→6。`metsuke rebuild` で全履歴に遡及。

## 根拠

1. **台帳自身の定義との整合**: 02 §3 で prompt は「人間の1プロンプト」。task-notification
   は人間の意思決定ではなく、「このターンに使うと人間が決めた」事実が存在しない。
   費用の意思決定主体は起点の指示だけである。
2. **ADR 0009 の意味論の完結**: 「この指示が引き起こした総コスト」には、agent走行費
   （ADR 0009）だけでなく「agentの報告を受けて main が行った後続作業」も因果的に含まれる。
   現状は因果ツリーが通知境界でぶつ切りになる。
3. **規模**（実運用台帳）: 通知プロンプトは全体の約8%だが、
   **総コストの約2割**が計上され、上位は多数requestを伴う規模
   （通知応答で次のagent波を起動するケース）。V1 period ビューのランキングに
   「指示の断片」が大量に混入する状態だった。
4. **情報の損失なし**: prompt 行と request.ts は不変のため、ターン単位の
   コスト（通知対応のオーバーヘッド）は必要なら再導出できる（ADR 0009 と同じ論理）。

## 条件と検証

- golden fixture に「通知プロンプト畳み込み（agent型・MCP型）」「task-id 不明の通知（温存）」
  「本文途中の引用通知（畳み込まない）」「通知ラリー中spawnの次波agentの推移的到達
  （固定点ループ必須の配置）」を追加し、増分==rebuild ハッシュ一致を維持。
- 実データ検証（rebuild 実施済み）:
  - v_daily の過去日総額が rebuild 前後で不変（帰属の移動のみ）。
  - 通知ラリーの費用が起点へ集約され、日次総額は不変だった。
  - 通知プロンプト残存は**約8割縮小**。残存の内訳は「tool-use-id も
    agent 連鎖も本文に無い旧形式通知」のみで、**解決可能なのに残存する件はゼロ**
    （tool-use-id はあるが tool_call 行不在の少数例を含む）。
  - 通知の頭アンカー判定は真の通知を全件拾い、引用だけの例を正しく除外した（偽陰性ゼロ）。

## 棄却した代替案

- **リンク列（prompt.origin_prompt_id）＋ビュー側集約**: ターン粒度が既定で残る利点は
  あるが、全ビュー・週次レポ・trace に二重意味論が入る（ADR 0009 で棄却した構図と同じ）。
- **現状維持**: 総コストの18.6%が「人間が発していないプロンプト」に帰属したままとなり、
  指示単位の意思決定（V1）が成立しない。
