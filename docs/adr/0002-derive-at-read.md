# ADR 0002: 事実のみ永続化し、金額・派生指標は読み出し時に導出する

日付: 2026-07-17 / 状態: 採択

## 決定

- 永続化するのは**事実**（トークン量・構造・イベント・人間とAIの判断）と**生ログ原本**のみ。
- コスト金額・実効トークン・恒等式破れ・ペース・タスク効率などの解釈は全て**SQLビュー**
  （git管理）で読み出し時に導出する。単価は SCD2 版管理テーブル（valid_from/to・
  5m/1h書込倍率・batch/fast/geo係数）とのJOINでのみ金額になる。
- `metsuke rebuild` により archive 原本から全派生物を冪等再構築できることを設計保証とする
  （rebuild の決定性と判断データの生存はテストで固定 — Roadmap Stage 1-7 / 4-2）。
- archive/ は**無リダクションの原本**とする。秘匿情報リダクションは読み出し境界
  （parse層・AI可視アクセサ）で versioned manifest により適用する — 誤検出しても
  再parseだけで回復でき、原本の不可逆破壊が起きない（[01 セキュリティ](../01-architecture.md)）。

## 根拠

1. **単価は自前管理**（APIで取得不能・請求突合で校正し続ける）という制約下では、
   金額を焼き込むと校正のたびに過去が誤ったまま残る。導出なら SCD2 行の追加だけで
   **全履歴が遡及再計算**される。
2. 指標定義・タスク分類器は必ず後から改良される。焼き込みは「定義変更＝過去と比較不能」を意味し、
   長期比較（J4）と施策の遡及再判定（J3）を壊す。
3. 個人規模（圧縮後は月あたり数十〜数百MB程度）では全量再計算が現実的。スケールのための増分パイプライン
   （＝沈黙故障の温床）を持つ必要がない。

## 注意（批評で特定された罠）

- **LLMによる分類（task_label）は純粋なビューでは実装不能**。分類結果は「AI提案→承認→事実として
  追記」の判断データとして扱い、分類器改良時は全履歴に再適用する（導出の精神を運用で保つ）。
- statusline/hooks 用の rollup・state.json は**使い捨てキャッシュ**と定義し、真実を持たせない。
  分析ビューとの数値不一致はバグとして扱う（同一のSQL定義から生成する）。
- 校正は**二段分離**（未計上を差し引いてから単価校正）。混同すると全履歴を誤単価で汚染する。

## 棄却した代替案

- 取込時のコスト焼き込み: 上記の遡及性を失う。
- 増分マテリアライズド集計: 増分の正しさというバグクラスを持ち込む。個人規模では不要。
