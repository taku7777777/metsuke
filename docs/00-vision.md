# 00 — ビジョン: 何のための道具か

## 目的関数

**LLM利用の行動変容**。利用者が設定した日次・週次・月次のヘッドルーム内で、
「同じ成果をより安く」または「同じ予算でより多くの成果を」実現する行動
（セッション運用・委任・モデル選択・依頼の仕方）が実際に変わること。

- 表示・集計・分析はすべて手段。**「見たが行動が変わらない」は失敗**と定義する。
- ヘッドルームは使い切ってよい予算であり、最小化目標ではない。**節約しすぎて手戻りが増える
  「過剰最適化」も失敗**であり、counter-metrics（手戻り率・上位モデル差し戻し率）で監視する。

## ジョブ（J1〜J8）

| # | ジョブ | 一言で |
|---|---|---|
| J1 | 単価感覚 | 1プロンプトがなぜこの値段かを**構造**（リクエスト列・ツール・系譜）で説明でき、着手前に見積もれる |
| J2 | 日次ペーシング | 今日は使いすぎか（同曜日比・着地予測）が常時視界にあり、暴走は**実行中に気づける**（止めるのは人間） |
| J3 | 週次最適化サイクル | 診断→施策1本→前後検証が毎週回り、施策の勝敗台帳が蓄積する |
| J4 | タスク単位の長期比較 | 全履歴でタスク種別ごとの効率推移が見える（保持期間の制約で諦めない） |
| J5 | 実行構造の理解 | 系譜ツリー・キャッシュ恒等式の破れ・cache_creation急増の因果をトレース状に探索できる |
| J6 | 計測ヘルス | 未計上・単価ズレ・取込停止・ラベル欠落を**システム自身が申告**する |
| J7 | AIアナリスト運用 | 週次の診断・帰属・施策提案・効果判定を Claude が回し、人間は読んで承認するだけ |
| J8 | 行動変容 | 気づきが行動の瞬間（送信前後・離席前・再開時）に届き、**従ったかまで計測**される |

| # | レバー | 動かすもの |
|---|---|---|
| 1 | コンテキスト量 | 取り込み・/compact・セッション分割 |
| 2 | キャッシュ継続性 | 放置・TTL失効後の重い再開・設定/モデル切替のタイミング |
| 3 | 委任と委任モデル | サブエージェント活用・fan-out 体数・委任先の単価 |
| 4 | モデル選択 | タスク難易度との整合 |
| 5 | effort | 思考量と行動量の水準 |
| 6 | 依頼の仕方 | 完全指定・中断/追い打ち・着手前見積り |
| 7 | 放置と背景機能 | 待機時間・background コスト |

## 設計原則

1. **事実は永久に、解釈は保存しない** — 生ログ不変アーカイブ＋読み出し時導出。
   未来の質問（新指標・単価改定・分類器改良）に全履歴で答えられる。
2. **介入は意思決定の瞬間に、探索は迷わない入口に配線する** — 送信前(hooks)・常時視界
   (statuslineの高コスト色と詳細リンク)・放置中の緊急通知に加え、deliberate層はコマンド知識を
   要求しないローカルdashboardから多角的に探索できるようにする（[ADR 0011](adr/0011-local-dashboard.md)）。
3. **沈黙故障を構造で殺す** — 鮮度は「最後に正常パースしたイベントの時刻」で定義し、
   毎日必ず見る場所(statusline)に出す。未知の値はNULL＋警告で顕在化（静かに間違うより煩く正しい）。
4. **AIが第一級の利用者** — 口はSQL＋スキーマ契約書＋`--json` CLI。人間とAIが同じ道具で同じ数字を見る。
   ただしAIの書き込みは提案止まり（人間承認ゲート）。
5. **保守者は本人＋Claude、一人** — 常駐プロセス最小・依存最小・全構成をas-codeでgit管理。
   壊れたら「気づける・直せる・遡れる」を、壊れない事より優先する。

## 非目標（意図的に捨てるもの）

| 捨てるもの | 理由 |
|---|---|
| 汎用・外部公開ダッシュボード基盤（Grafana/Loki/Prometheus/Tempo等） | 保存・監視・外部公開・運用面積が本目的に過剰。read-onlyの小さなローカルUIは対象外（ADR 0011） |
| 既製LLM可観測性（Langfuse/Phoenix/LangSmith/OpenLIT） | マルチテナント・eval基盤・SDK計装前提が過剰。施策効果検証・実請求突合という本システムの核をどれも持たない。輸入するのはデータモデルのパターンのみ |
| 厳密な統計的効果検証（A/B・有意差判定） | 単一ユーザーは統制実験にならない。施策マーカー＋前後マクロ比較＋AI叙述に割り切り、代わりに遡及再判定可能性で補強 |
| 秒未満のリアルタイム / ストリーミング基盤 | J2の要件は〜1分。hook駆動（通常経路で60秒以内）＋launchd 5分フォールバックで足りる |
| コスト金額・派生指標の永続化 | 遡及再計算可能性と引き換えにできない |
| 自動介入（自動モデル切替・自動ブロック） | 誤爆1回で道具全体の信頼が死ぬ。警告＋ドル建て提示＋代替行動の名指しに徹する |
| マルチユーザー・HA・クラウド同期・外部SaaS送信 | 対象外（ローカルファースト・PII含有データ） |
| OTel GenAI semconv 準拠 | Development段階で不安定。消費者が本人とClaudeだけのスキーマに標準準拠の便益がない |

## 成功基準

- **行動**: ナッジのconversion率（提案後10分以内の行動転換）が計測され、無効ナッジが特定できる。
  cold-cache再構築費・context 150K超の往復数など「防げた浪費」が減少傾向を示す。
- **ROI**: システム自身の運用コスト（週次アナリスト＋自動ラベリング＋申告保守時間）< 検証済み削減額
  （四半期ごとに `metsuke roi --days 90` で確認）。**判定ルール: 2四半期連続で赤字なら週次アナリストを停止し
  Stage 2 構成（台帳＋statuslineの色/詳細導線＋緊急通知）へ縮退**する（[04-roadmap](04-roadmap.md)）。
- **生存**: 3ヶ月後も使われていること。手動の**データ入力儀式**は月1回（請求突合）以下に抑える。
